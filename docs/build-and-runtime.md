# Build and runtime map

This document explains how the custom build works in plain language.

## The short version

`npm run build` does two things:

1. Rollup builds the app into `.tmp/build`.
2. `lib/move-output.js` deletes the old `build/` folder and moves `.tmp/build/static` to `build/`.

`npm run dev` does two things at the same time:

1. Rollup watches source files and rebuilds `.tmp/build`.
2. `lib/dev-static-server.js` serves `.tmp/build/static` through `serve`.

`npm run preview` serves the finished `build/` folder through `lib/preview-server.js`. Use it after `npm run build` when testing the production output locally.

## TypeScript

The root `tsconfig.json` uses TypeScript project references:

- `client-tsconfig.json`
- `worker-tsconfig.json`
- `static-build-tsconfig.json`

All TypeScript output goes to `.tmp/ts`. Rollup does not directly compile TypeScript. Instead, `lib/simple-ts.js` runs `tsc -b` and then Rollup reads the generated JavaScript from `.tmp/ts`.

## Rollup

The Rollup entry point is `src/static-build/index.tsx`. That entry creates the static shell files, then the Rollup config uses custom plugins to bundle the browser app, workers, CSS, assets, and service worker.

Important custom plugins:

- `lib/feature-plugin.js`: scans `src/features/` and generates feature wiring files.
- `lib/client-bundle-plugin.js`: creates the browser-side JavaScript bundles.
- `lib/sw-plugin.js`: builds the service worker.
- `lib/css-plugin.js`: handles CSS modules and emitted CSS.
- `lib/url-plugin.js` and `lib/data-url-plugin.js`: handle imported assets.
- `lib/run-script.js`: runs the generated static-build script so it writes HTML/manifest/header files.

Important local server wrappers:

- `lib/dev-server.js`: starts `npm run watch` and `npm run serve` together, then cleans up both child processes on exit.
- `lib/dev-static-server.js`: starts the `serve` package against `.tmp/build/static` using the repo's `serve.json`.
- `lib/preview-server.js`: starts the `serve` package against the finished `build/` directory using the repo's `serve.json`.

## Generated files

These source files are generated by the build:

- `src/client/lazy-app/feature-meta/index.ts`
- `src/client/lazy-app/feature-meta/shared.ts`
- `src/client/lazy-app/feature-meta/encoders.ts`
- `src/client/lazy-app/worker-bridge/meta.ts`
- `src/client/lazy-app/worker-bridge/surface.ts`
- `src/features-worker/index.ts`

They are important because the app discovers features by folder convention
rather than by a hand-written registry. The generated compatibility outputs are
ignored by Git unless a file is intentionally tracked as a source seam.

## Client app parts

The UI is split to make the first page load smaller:

- Initial app: `src/client/initial-app/`

  - Loads quickly.
  - Shows the intro.
  - Handles file drop/pick and basic routing.
  - Lazy-loads the editor.

- Lazy editor: `src/client/lazy-app/`
  - Decodes the input image.
  - Runs processors and encoders.
  - Shows the side-by-side comparison.
  - Handles downloads and saved side settings.

## Worker bridge

The browser cannot directly call functions inside a Web Worker. This project uses Comlink to make worker calls look like normal async function calls.

`src/client/lazy-app/worker-bridge/index.ts` starts a worker when needed, queues work, and shuts the worker down after it has been idle for 10 seconds.

The actual worker methods come from feature files under `src/features/**/worker/`.

## Service worker

`src/sw/index.ts` controls offline caching and the PWA share target.

Key behavior:

- caches basic app files during install;
- caches more processor/codec files after the user interacts with the app;
- redirects `/editor` back to `/` when loaded directly;
- handles image files shared into the installed PWA;
- removes old versioned caches during activation.

Because service workers cache aggressively, production bugs can look different from local dev bugs. Test with a fresh browser profile or clear site data when debugging cache behavior.

## Codec outputs

The `codecs/` folder contains codec projects and prebuilt outputs. The app imports the generated JavaScript and WebAssembly files from these folders.

Common codec output files:

- `.wasm`: the compiled codec binary.
- `.js`: JavaScript glue code that loads and calls the WASM.
- `.d.ts`: TypeScript declarations for the codec API.

Do not delete codec outputs unless you are ready to rebuild and retest the corresponding feature.
