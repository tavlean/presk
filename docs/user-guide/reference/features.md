# Frisp editor — feature reference

Exhaustive, code-derived inventory of every feature and user interaction in the
editor and bulk mode. Each entry lists the source file and one precise sentence
on what it does. Derived purely from reading the source — see "Notable / hidden
behaviors" at the end for things not surfaced in the obvious UI.

## Loading & navigation

| Feature                                | File                                                     | What it does                                                                                                                                                                                                       |
| -------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Intro landing screen                   | `src/lib/editor/intro/Intro.svelte`                      | Shown while no file is loaded (the `{:else}` branch of `+page.svelte` renders `<Intro>`): a full-viewport dashed viewfinder frame with the chrome reduced to HUD micro-copy pinned to its inner corners (brand lockup top-left, the format list "WebP · AVIF · JPEG XL · JPEG · PNG" bottom-left, "offline · open source · private" and "Frisp — images never leave your device" bottom-right); the centre column holds a tray icon, the headline "Drop images anywhere." (swaps to "Release to add." while dragging, "Add an image." on a coarse-pointer/touch device), a subline, a "Browse files" button, and a quiet "or choose a folder · paste" row. Themes off the OS via CSS `light-dark()`, no manual toggle. |
| Browse / folder / paste actions        | `src/lib/editor/intro/Intro.svelte`                      | A "Browse files" `<button>` clicks a hidden multi-file `<input type="file" accept="image/*" multiple>`; a separate hidden `webkitdirectory` input (no `accept`) backs the quiet "choose a folder" link; both inputs' `onchange` runs `onPick`, which sends the files up through `routeFiles` (2+ supported images open bulk, 1 opens the single editor). A "paste" link (async clipboard read) and window ⌘/Ctrl+V paste add clipboard images the same way.                                                                                          |
| Drag-and-drop to load/replace          | `src/lib/editor/file-drop.ts`, `src/lib/editor/intro/Intro.svelte`, `src/routes/+page.svelte` | Dropping images anywhere loads or replaces the current file instead of letting the browser open them: `.app-root` carries the `fileDrop` attachment for the open editor, while the intro attaches its own drop target (with `stopPropagation`) so a landing drop routes exactly once. Both paths land in `routeFiles`, and dropped folders are walked recursively.    |
| Drag-over visual feedback              | `src/lib/editor/file-drop.ts`, `src/lib/editor/intro/Intro.svelte`, `src/routes/+page.svelte` | Over the open editor, a file-bearing drag toggles a `drop-valid` class on `.app-root`, fading in a coral dashed full-viewport overlay (an enter/leave depth counter prevents flicker over nested children). Over the intro the coral overlay never shows: the intro stops propagation and instead lights its own dashed viewfinder frame (accent stroke plus inner glow) as the drag feedback.                    |
| Files-only drag filter                 | `src/lib/editor/file-drop.ts`                            | `dragHasFiles` ignores drags that carry only text/links (no `Files` type), so text drags don't trigger the overlay or `preventDefault`.                                                                            |
| Non-image rejection                    | `src/routes/+page.svelte` (`routeFiles`), `src/client/lazy-app/bulk/import.ts` (`isSupportedBulkImage`) | Dropped/picked files are filtered by `isSupportedBulkImage` (by extension); if none qualify, `routeFiles` shows the snackbar `No supported images found.` and loads nothing. A secondary MIME guard in `pickFiles` (`editor-session.svelte.ts`) still exists — `"<name>" doesn't look like an image.` — for a file that clears the extension filter but isn't an image. |
| Open pushes history state              | `src/routes/+page.svelte`, `editor-session.svelte.ts`    | Opening the first image calls `pushState('', { editor: true })` so the browser Back button (or gesture) returns to the intro screen.                                                                               |
| Route-state sync / Back closing editor | `editor-session.svelte.ts` (`syncRouteState`)            | An `$effect` watches `page.state.editor`; when it becomes false (Back navigation) while a file is loaded, `clearFile()` is called, returning to the intro.                                                         |
| Back button                            | `src/routes/+page.svelte` (`back`)                       | The round glass "X" button top-left calls `history.back()` (falling back to `clearFile()` if no history), unwinding the pushed editor state.                                                                       |
| Undo / Redo buttons                    | `src/routes/+page.svelte`, `editor-session.svelte.ts` (`undo`/`redo`) | Two glass buttons right of Back step the editor document backward/forward through its history; each is `disabled` via `history.canUndo` / `history.canRedo`. Tooltips show ⌘ vs Ctrl by platform.    |
| Undo / Redo keyboard shortcuts         | `src/routes/+page.svelte` (`onKeydown` on `<svelte:window>`) | `⌘/Ctrl+Z` undoes, `⇧⌘Z` / `Ctrl+Shift+Z` / `Ctrl+Y` redoes. Suppressed when a typeable field (text/number input, textarea, contenteditable) is focused so native text-undo still works; range/checkbox inputs fall through to editor undo. |
| Document title                         | `editor-session.svelte.ts` (`docTitle`)                  | The tab title is `"<filename> - Frisp - Compress images"` (`APP_NAME` = "Frisp"), prefixed with an ⏳ hourglass while either side is encoding.                                                                     |
| Dev-only diagnostics link              | `src/routes/+page.svelte`                                | In dev builds only, a "Pipeline diagnostics →" link to `/diagnostics` appears on the intro screen.                                                                                                                 |

## Two-up before/after compare (Output)

| Feature                           | File                                                                           | What it does                                                                                                                                                                                                                                         |
| --------------------------------- | ------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Two-up split view                 | `src/lib/editor/output/Output.svelte`, `output/two-up.ts`, `output/two-up.css` | A `<two-up>` custom element overlays the left side (canvas 0, "before") and right side (canvas 1, "after"), clipping each to one side of a movable split point.                                                                                      |
| Left = before / right = after     | `Output.svelte`, `src/routes/+page.svelte`                                     | Left canvas draws `session.results[0].outputImageData` (side 0's output, accented coral), right draws side 1's output (accented azure); both default to side 0=Original, side 1=WebP.                                                                |
| Draggable split handle            | `output/two-up.ts`, `output/two-up.css`                                        | A `.two-up-handle` with a circular glass scrubber (coral ◀ / azure ▶ arrows) is dragged via `PointerTracker`, clamped to `[0, bounds]`, updating the `--split-point` CSS var that drives both clip-paths.                                            |
| Relative split position retained  | `output/two-up.ts` (`_resetPosition`)                                          | The handle's position is stored as a fraction (`_relativePosition`, default 0.5) and re-derived on every resize via a `ResizeObserver`, so the split stays proportional when the window changes.                                                     |
| Orientation (horizontal/vertical) | `Output.svelte` (`orientation` derived), `two-up.css`                          | The split is vertical (top/bottom) when the viewport is ≤760px wide and horizontal (left/right) otherwise; the handle, clip-paths, and scrubber rotation all switch accordingly.                                                                     |
| Keyboard split presets (1/2/3)    | `output/two-up.ts` (`_onKeyDown`)                                              | Pressing `1`/`2`/`3` (Digit or Numpad) snaps the split to start, middle, or end — but is ignored while focus is inside an `<input>`.                                                                                                                 |
| Synced pan/zoom across sides      | `Output.svelte` (`onLeftChange`), `output/pinch-zoom.ts`                       | Only the left `<pinch-zoom>` is driven; its `change` event mirrors scale/x/y onto the right `<pinch-zoom>` so both halves stay perfectly aligned.                                                                                                    |
| Event retargeting to left side    | `Output.svelte` ($effect with capture listeners)                               | Touch/mouse/wheel (+ pointerdown on non-Safari) events on the two-up are cloned and re-dispatched to the left pinch-zoom; drags over the handle pass through (so the slider works) but wheel always retargets so zooming works even over the handle. |
| Pinch-zoom (touch)                | `output/pinch-zoom.ts` (`_onPointerMove`)                                      | Two-finger pinch scales about the gesture midpoint and pans by the midpoint delta; one-finger drag pans the image.                                                                                                                                   |
| Auto fit + centre on load/resize  | `Output.svelte` ($effect with `fittedKey`)                                     | On a new file, dimension change, orientation flip, or viewport resize (keyed, so re-encodes at the same size don't refit), the image is scaled to fit (never above 1×) and centred inside the inset-adjusted viewport via a `requestAnimationFrame`. |
| Fit insets                        | `Output.svelte`, `src/routes/+page.svelte` (`--fit-inset-*`)                   | Desktop reserves 300px on each side (`--fit-inset-left/right`) so the fitted image clears the two option panels; on mobile (≤760px) these insets collapse to 0.                                                                                      |
| Contain-letterbox alignment       | `Output.svelte` (`containStyle`), `editor-session.svelte.ts` (`sideContains`)  | When a side uses Resize with fit-method "contain", its canvas CSS box is pinned to the original source dimensions with `object-fit:contain`, so a smaller output stays letterboxed inside the original footprint and the two halves remain aligned.  |
| Vector-true SVG preview           | `src/routes/+page.svelte`, `output/SvgPreview.svelte`                          | When a side shows the optimized `svg` output, an `<img>` overlay (`data-pinch-overlay`) re-rasterises the SVG at every zoom level — its layout width/height and `translate` mirror the driven pinch-zoom transform instead of sitting under it — so the vector stays crisp at any magnification; past 8192px on the long edge the residual zoom is applied as a bitmap `scale`. |

The resize-vs-other call diffs each side's _effective_ resize recipe (resize counts only when target dims ≠ source) against the previous pass. |
| Redundant-pass skip          | `editor-session.svelte.ts` (`encodeSide`, `displayedSig`)                    | A side re-encodes only when its effective request (format, options, preprocessor, quantize, and resize **when it changes the size**) differs from what's on screen (`displayedSig`) **and** isn't already in the shared result cache. So enabling Resize at 100%, or toggling Premultiply/Linear RGB at 100%, is a true no-op — no encode, no pill — and any previously-seen request is served from cache instead of re-encoding. `processImage` independently skips the resampling call when target dims equal the source.                 |
| Type label (mobile)          | `Results.svelte`                                                           | On ≤760px the short format label (e.g. "WebP") is shown next to the size.                                                                                                                                                                                                                                                                                                                                                   |
| SVG gzip size line           | `Results.svelte`, `compress.ts` (`svg`), `src/lib/svg/optimize.ts`         | On the SVG lane the stats add a second line "gzip: <n> · was <n>" — the optimized text's gzip size vs the source's — because gzip is what an SVG actually costs over the wire and the raw-byte figure understates the win. The raw output never regresses: `keepOriginalSvg` reverts to the source text (0% change) if the optimized bytes aren't smaller.                                                                       |

## Encoding pipeline & lifecycle

| Feature                  | File                                                                 | What it does                                                                                                                                                                                                                                                          |
| ------------------------ | -------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Per-side reactive encode | `src/routes/+page.svelte`, `editor-session.svelte.ts` (`encodeSide`) | An `$effect` per side re-encodes whenever its format/options/processor/preprocessor state changes, running decode→preprocess→resize→quantize→compress through the shared image pipeline (WASM codec worker) — unless the result is already on screen or in the cache (below).                                                          |
| Decode-back preview      | `compress.ts` (`compressFile`)                                       | The encoded output is decoded back to pixels for the "after" canvas so the preview shows the real codec result (artifacts included), aligned in size with the processed source.                                                                                       |
| Debounced re-encode      | `editor-session.svelte.ts` (`IMAGE_UPDATE_DELAY`)                    | Edits debounce by 100ms before re-encoding (so dragging a slider doesn't spawn an encode per frame); a fresh file encodes immediately. A cache hit skips the debounce entirely (it's shown synchronously).                                                                                                                                |
| Result cache (instant revisit) | `src/lib/result-cache.ts` (`ResultCache`), `editor-session.svelte.ts` (`encodeSide`) | Finished `CompressOutcome`s are cached keyed by their full encode-input signature, **shared across both sides** (a result is purely a function of its inputs, not which side shows it). Revisiting a recipe — via Undo/Redo, a re-selected setting, or the other side matching — shows it instantly with no re-encode. An LRU evicts by a pixel-byte budget (default 256 MiB) + entry cap (24), keeping currently-displayed results pinned. |
| Abort in-flight encodes  | `editor-session.svelte.ts`                                           | Each encode uses an `AbortController`; a superseding change aborts the prior run and revokes its orphaned object-URL (it is not cached).                                                                                                                                                 |
| Object-URL lifecycle     | `result-cache.ts` (`ResultCache`), `editor-session.svelte.ts`       | Each cached result owns its download object-URL; the cache revokes it on LRU eviction and on `clear()` (new file, `clearFile`, dispose). Aborted/duplicate encodes revoke their own URL inline. Displayed results stay pinned so their URL is never revoked underfoot.                                                                                                                                                                   |
| Undo/redo history        | `src/lib/editor/editor-history.svelte.ts` (`EditorHistory`), `editor-session.svelte.ts` (`watchHistory`/`undo`/`redo`) | A debounced `$effect` watcher (`HISTORY_COMMIT_DELAY` 350ms) commits a snapshot of the document (`{ sides, preprocessorState }`) once edits settle, so a slider drag coalesces into one step. Snapshots are stored in `$state.raw` and de-duped by an output-affecting signature; a guard flag + that signature stop a restore from re-committing. Undo/redo restore a snapshot, which re-runs the encode effect into a cache hit. History resets per image (`resetHistory` on load) and clears on `clearFile`. |
| Per-side error pill      | `src/routes/+page.svelte`, `editor-session.svelte.ts` (`firstError`) | The first side error message is shown as a status pill at the top-centre of the editor.                                                                                                                                                                               |
| Settings persistence     | `editor-session.svelte.ts` (`persistSettings`)                       | Both sides' format + per-format options are saved to localStorage (`app:settings:v3`) on change and rehydrated on next load (resize/quantize processor state is NOT part of this global persistence — only the per-side Save/Import buttons store processor state). |
| Resize-dimension seeding | `editor-session.svelte.ts` (`seedResizeDimensions`)                  | Once the first result lands, the Resize width/height inputs are seeded with the preprocessed source dimensions on sides where resize is disabled.                                                                                                                     |
| SVG/vector handling      | `editor-session.svelte.ts` (`pickFiles`/`isVectorSource`/`encodeSide`), `src/lib/svg/optimize.ts` | An SVG source defaults side 1 to the `svg` optimize lane and each side's resize method to "vector" (the "Vector" method is only offered then) so re-rasterising stays crisp. When a side's format is `svg`, `encodeSide` dispatches to the lazy-imported `optimizeSvgSide` (SVGO + fflate stay out of the main bundle) instead of the raster pipeline; if the current source isn't SVG the `svg` format falls back to `webP`. |

## Notable / hidden / non-obvious behaviors

- **Keyboard split presets (1/2/3)** in `two-up.ts` jump the comparison slider to start/middle/centre but are undocumented in the UI and suppressed while typing in an input.
- **Wheel always retargets to the left pinch-zoom**, so scroll-to-zoom works even with the cursor over the drag handle (`Output.svelte` event retargeting).
- **Safari-specific paths**: the smoothing/pixelated row is omitted from the View options popover on Safari (`image-rendering: pixelated` is a no-op in WebKit), so on Safari the popover shows only the Background row (`Output.svelte`); `pointerdown` is excluded from the retargeted event list on Safari; and `pinch-zoom.ts` sets `avoidPointerEvents: isSafari` to work around an iOS WebKit pointer-capture bug (#220196).
- **Android keyboard blur**: on `touchend` over the output, the active element is blurred so Android Chrome doesn't re-pop the software keyboard (ported `shouldBlurActiveElementAfterOutputRetarget`).
- **Firefox focus workaround**: the editable zoom input forces a style recalc (`getComputedStyle().transform`) before `focus()`/`select()` so Firefox doesn't drop focus on the just-inserted input (`focusOnMount`).
- **Quantize "ZX" konami easter-egg is intentionally OMITTED** from the Svelte port — `QuantizeOptions.svelte` notes the original's hidden konami-code palette type was dropped.
- **Trackpad pinch vs wheel**: `pinch-zoom.ts` distinguishes trackpad pinch (`ctrlKey`) from a plain wheel by using a tighter zoom divisor, and scales "lines"-mode deltas ×15.
- **Inverted-value sliders**: several "Effort"/"Slight loss"/"Filter sharpness" sliders display the inverse of the stored codec value (e.g. WebP near_lossless, filter_sharpness; AVIF effort=10−speed), so the displayed scale is friendlier than the raw option.
- **Lossless inference**: AVIF/JXL panels infer the Lossless checkbox state from the underlying quality/subsample rather than a dedicated stored flag, and JXL force-enables "Alternative lossy mode" (disabled) below quality 7.
- **Save vs global persistence asymmetry**: the global auto-persist (`app:settings:v3`) stores only format + options (not processor/resize/quantize state), whereas the per-side Save button (`app:side-settings:*`) stores the full state including processor state — so a reload restores encoder choices but not resize/quantize toggles.
- **Import validation gating**: the Import button's enabled state and the import action share one `parseSavedSide` validator, so a corrupt or partial saved payload both disables the button and (if forced) shows an "invalid" snackbar — they can't drift.
- **`legacy-clip-compat` mode**: `<two-up>` positions its children absolutely and uses `clip-path` insets (rather than transform-based clipping) for cross-browser compatibility.
- **Result cache is shared across sides**: the cache key is the encode-input signature with **no side index**, because an encode is a pure function of its inputs — so a result the right side computed loads instantly if the left side later matches it (verified: ~20ms cached return vs ~470ms re-encode). Undo/redo and toggling a setting back are the same mechanism: a cache hit, not a re-encode.
- **Undo history excludes no-op state**: the history signature uses the active format's options and folds a disabled resize to `null`, so seeding the (hidden) resize dimensions after the first encode doesn't spawn a phantom undo step — mirroring the encode-skip logic.
- **Identity/Original download is the raw file**: the Original side's download is `URL.createObjectURL(file)` of the untouched original (0% change), not a re-encode.
- **No-JS / SSR safety**: `localStorage` access is guarded (`canUseLocalStorage`) so private-browsing/SSR don't throw; settings simply don't persist there.
